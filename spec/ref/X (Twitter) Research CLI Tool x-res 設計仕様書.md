# X (Twitter) Research CLI Tool: `x-res` 設計仕様書

本ドキュメントは、XのAPIを駆使して高度なソーシャルリサーチを行うための、AI機能を持たない純粋なデータ収集・加工に特化したコマンドラインツール（CLI）の設計・仕様書です。

## 1. コンセプトと設計思想

- **UNIX思想の踏襲:** 1つのコマンドは1つのことに集中する。フィルタリング、ソリューション、JSON/CSVでのクリーンな出力を行い、パイプライン（`jq` や後続のAIスクリプト、BIツール）とシームレスに連携する。
    
- **APIレートリミット（制限）の徹底管理:** X APIで最もボトルネックとなるレートリミットを自動追跡し、限界に達した場合は安全にスリープ（ウェイト）または進捗をローカルにキャッシュして再開できる堅牢性。
    
- **非同期・並行処理:** 大量データ取得時のパフォーマンスを最大化するため、非同期I/O（Async/Await）による並行リクエスト処理を採用。
    

## 2. コマンド構成 (CLI CLI Commands)

ツール名を `x-res` (X-Research) とし、5つのリサーチ優先度に対応するコマンドを提供します。

```
x-res <command> [options]
```

### ① `x-res trend`（トレンド・予兆検知）

指定したキーワード群の「出現頻度の急増（スパイク）」や「共起語」を抽出します。

- **出力例:** 1時間ごとのツイート件数推移（時系列データ）、同時によく呟かれている単語（共起語）のランキング。
    

### ② `x-res voice`（顧客の本音分析）

ノイズ（PRポスト、BOT、アフィリエイト、相互フォロー募集など）を徹底的に除外し、生の購買・利用体験に関するポストのみをフィルタリングして抽出します。

- **出力例:** 除外フィルター適用済みのクリーンなポスト一覧（テキスト、ユーザーID、エンゲージメント数）。
    

### ③ `x-res network`（コミュニティ・インフルエンサー解析）

指定したポストのリポスト関係、または特定アカウントのフォロワーネットワークを解析し、グラフデータベースや可視化ツール（Gephiなど）に読み込める形式で出力します。

- **出力例:** エッジ（送信元 -> 送信先）とノード（ユーザー情報、次数）のCSVリスト。
    

### ④ `x-res benchmark`（競合比較・SoV）

複数のキーワード（ブランド名等）を指定し、それぞれの言及ボリュームやエンゲージメント率の推移を並列で取得して比較します。

- **出力例:** ブランドごとの日次投稿数、総いいね数、平均リポスト数のマトリクス。
    

### ⑤ `x-res archive`（長期時系列推移）

過去の特定期間（数ヶ月〜数年単位）を指定し、月次・週次レベルでの大まかな言及ボリュームの変化をサンプリング取得します。

## 3. 詳細なコマンド仕様とオプション

### 1. `trend` コマンド

```
x-res trend --query "生成AI" --interval 1h --limit 5000 --output out.json
```

- **オプション:**
    
    - `-q, --query <string>`: 調査対象のキーワード（論理演算 `AND/OR/NOT` 対応）
        
    - `-i, --interval <1m|10m|1h|1d>`: 集計の最小時間単位
        
    - `--co-occurrence`: 共起語分析を有効化し、上位N個の共起単語リストを併せて出力
        
- **処理ロジック:**
    
    1. `GET /2/tweets/search/all` または `recent` を呼び出す。
        
    2. ツイート本文を簡易形態素解析（日本語の場合は軽量なライブラリや正規表現による簡易トークナイズ、またはユーザーの外部システムに委ねる場合はストップワード除外のみ）し、共起頻度を算出。
        

### 2. `voice` コマンド

```
x-res voice --query "ルンバ" --clean --min-likes 5 --output feedback.csv
```

- **オプション:**
    
    - `--clean`: スパム、BOT、PR投稿（`PR`, `ad`, `アフィリエイト`, `相互` などの文字列、過剰なハッシュタグ、怪しい短縮URL）を正規表現パターンで自動除外。
        
    - `--min-likes <number>` / `--min-retweets <number>`: 一定以上のエンゲージメントがあるポストに絞り込み（ノイズ対策）。
        
    - `--exclude-retweets`: リポスト（RT）は除外し、オリジナルポストのみを対象とする。
        

### 3. `network` コマンド

```
x-res network --tweet-id "1234567890" --depth 2 --output network.gexf
```

- **オプション:**
    
    - `--tweet-id <string>`: 起点となる投稿ID。この投稿をリポスト・引用したユーザーを起点にネットワークを構築。
        
    - `--username <string>`: 特定のインフルエンサーを起点に、そのフォロワー同士の関係性を追跡。
        
    - `--depth <1|2>`: 探索するネットワークの深さ（API制限を考慮しデフォルトは1、最大2推奨）。
        
    - `-f, --format <csv|json|gexf>`: グラフ可視化ソフトで一般的な `.gexf` 形式の出力に対応。
        

### 4. `benchmark` コマンド

```
x-res benchmark --targets "メルカリ,ヤフオク,ラクマ" --span 30d --output compare.csv
```

- **オプション:**
    
    - `--targets <comma-separated-strings>`: 比較したい複数のブランド・キーワード（例: `A,B,C`）。
        
    - `--span <7d|30d|90d>`: 比較する期間。
        
- **処理ロジック:**
    
    - ターゲットごとに非同期で検索APIを実行。
        
    - 同一時間軸上のボリューム（Share of Voice）およびエンゲージメント特性（「Aはいいねが付きやすいがリポストはBの方が多い」など）を横並びの表形式で出力。
        

## 4. システムアーキテクチャ & 設計図

```
[User CLI Input] ──> ( x-res コマンド解析 )
                           │
                           ▼
                  [ Rate Limit Manager ] <─── (ローカルDB: SQLite/File)
                  (API残リクエスト数・復旧時刻管理)   ※中断時の進捗もキャッシュ
                           │
                           ▼
                 [ Async API Client ] (HTTP / v2 API)
                 (並行リクエスト・指数バックオフ再試行)
                           │
       ┌───────────────────┴───────────────────┐
       ▼                                       ▼
 [ GET /tweets/search ]                  [ GET /users/id ] 等
       │                                       │
       └───────────────────┬───────────────────┘
                           ▼
                [ Data Processor / Filter ]
                (スパム除外、共起語抽出、グラフノード作成)
                           │
                           ▼
                    [ Formatter ]
             (CSV / JSONL / GEXF 変換) ──> [ 標準出力 or ファイル ]
```

## 5. 開発における重要技術仕様

### ① レートリミットハンドリング (超重要)

X APIはリクエスト制限が非常に厳しいため、以下の機構を組み込みます。

- **ヘッダー情報の常時監視:** レスポンスヘッダーに含まれる `x-rate-limit-remaining`（残数）と `x-rate-limit-reset`（復旧時刻）を解析。
    
- **自動待機（Cool-down）モード:** 残数がゼロになった場合、プログラムを異常終了させず、復旧時刻まで `sleep` して処理を自動再開するオプション `--wait-on-limit` を搭載。
    
- **レジューム（再開）機能:** 大量データ取得中にエラーや中断が発生した場合、取得済みの最新の `since_id` や `next_token` をローカルテンポラリファイル（またはSQLite）に保存し、次回実行時に `--resume` オプションで続きから取得可能にする。
    

### ② 入出力データの標準化 (データパイプライン化)

出力形式はデフォルトで **JSONL (JSON Lines)** とします。

- **理由:** 大容量データを1行1JSONオブジェクトとして出力することで、全体のメモリ（RAM）を圧迫せずにストリーム処理でファイルに書き出すことができ、後続の `grep` や `jq`、Pythonスクリプトによる1行ずつの高速パースが可能になるためです。
    

```
{"id":"1928374...", "created_at":"2026-06-07T14:20:00Z", "text":"本音テキスト...", "user":{"id":"9876...", "followers_count":1200}, "metrics":{"like_count":12, "retweet_count":2}}
{"id":"1928375...", "created_at":"2026-06-07T14:21:00Z", "text":"本音テキスト2...", "user":{"id":"5432...", "followers_count":350}, "metrics":{"like_count":5, "retweet_count":0}}
```

### ③ エラーハンドリングとリトライ戦略

API呼び出しの接続エラーや一時的な 5xx エラーに対処するため、以下を実装します。

- **指数バックオフ（Exponential Backoff）:** エラー検知時に `1s -> 2s -> 4s -> 8s -> 16s` と段階的に待機時間を増やしながら最大5回までリトライする。
    

## 6. この設計がもたらす価値

このCLIアプリは「AI機能」をあえて内蔵していません。それにより、以下のような強固なメリットが生まれます。

1. **軽量・ポータブル:** 重い機械学習ライブラリや巨大な依存関係がなく、あらゆるサーバーやローカルPC（Mac/Windows/Linux）で即座に動作する。
    
2. **インフラの分離:** APIキーの管理やスクレイピング/収集部分だけをこのCLIに任せ、重い「AI感情分析」や「要約」は、収集したJSONLファイルを読み込む別のスクリプト（Pythonやクラウド上のLambda/Cloud Run等）に分離できる。
    
3. **自動化の容易さ:** Linuxの `cron` やGitHub Actionsなどのタスクスケジューラーに組み込み、毎日深夜に自動で競合データを取得してデータベースにストックする、といった運用が容易になる。
